第一阶段：设计与选型 (第 1 周)
目标：完成架构设计，避免开发中途推倒重来。

 1.1 确定业务边界
 核心功能：用户登录、资源管理（如球场/诊室）、时间段管理、预约接口、取消预约。
 砍掉功能：不要做复杂的前端页面（用 Swagger/Postman 代替）、不要做支付对接（模拟成功即可）、不要做复杂的后台管理（简单 CRUD 即可）。
 交付物：一份功能列表文档（Markdown）。
 1.2 技术栈定型
 JDK: 17 或 8 (根据公司要求，建议 17 体现学习力)
 框架：Spring Boot 3 + MyBatis-Plus
 数据库：MySQL 8.0
 缓存：Redis 7.0 (集群模式更佳)
 消息队列：RabbitMQ 或 RocketMQ (推荐 RocketMQ 事务消息)
 锁/协调：Redisson
 网关/限流：自研 Spring Boot Starter 或 Spring Cloud Gateway
 交付物：pom.xml 依赖清单。
 1.3 数据库设计 (ER 图)
 用户表 (user)
 资源表 (resource, 如球场)
 时段库存表 (resource_stock, 核心表：资源 ID+ 日期 + 时段 + 库存)
 预约记录表 (reservation, 包含状态机字段)
 关键点：设计唯一索引防止重复预约 (user_id + resource_id + date + slot)。
 交付物：SQL 建表脚本 + ER 图。
 1.4 环境搭建
 安装 Docker, MySQL, Redis, MQ。
 初始化 Git 仓库，配置 .gitignore。
 交付物：可运行的空白 Spring Boot 项目。
🛠️ 第二阶段：核心业务开发 (第 2-3 周)
目标：完成基本流程，确保业务逻辑跑通。

 2.1 基础架构搭建
 统一返回结果封装 (Result<T>)。
 全局异常处理 (@ControllerAdvice)。
 日志规范 (SLF4J + Logback，区分 info/error)。
 交付物：通用工具类包。
 2.2 用户与认证
 实现注册/登录。
 集成 JWT 或 Spring Security 进行接口鉴权。
 交付物：登录接口可通，携带 Token 访问。
 2.3 资源与库存管理
 管理员接口：创建资源、初始化时段库存。
 用户接口：查询可预约时段。
 交付物：后台可配置库存，前台可查。
 2.4 预约主流程 (同步版)
 实现最基础的 reserve() 方法。
 逻辑：查库存 -> 扣库存 -> 建订单。
 注意：此时先不加锁，为了后续优化做对比基准。
 交付物：接口可成功创建预约记录。
🚀 第三阶段：高并发核心攻坚 (第 4-5 周) ⭐⭐⭐⭐⭐
目标：这是你简历的核心亮点，必须深究。

 3.1 引入 Redis 缓存
 缓存热点资源信息（减少 DB 读）。
 缓存时段库存（减少 DB 读）。
 解决缓存一致性：采用 Cache Aside 模式，更新 DB 后删缓存。
 交付物：接口响应时间降低 50% 以上。
 3.2 解决超约问题 (锁)
 方案对比：先写数据库乐观锁版本，再写 Redis 锁版本。
 Redis 锁实现：使用 Redisson RLock。
 锁粒度优化：锁 Key 设计为 lock:reserve:resourceId:date:slot (细粒度锁)。
 看门狗机制：确保业务执行时间长于锁过期时间时不失效。
 交付物：并发测试下库存不为负数。

 3.3 异步削峰 (MQ)
 将“扣库存 + 写订单”改为同步，后续动作（如发短信、积分）改为异步。
 或者：请求先入 MQ，消费者慢慢写库 (极致削峰)。
 交付物：高峰期接口不超时，MQ 无消息堆积。
 3.4 幂等性设计
 防止用户重复点击提交。
 实现：前端生成 Token -> 后端校验并删除 Token (Redis 原子操作)。
 交付物：重复请求返回相同结果，不产生重复订单。
 3.5 自研风控网关 (集成)
 实现基于 Redis Lua 的 滑动窗口限流算法。
 接口注解 @RateLimit(limit=10, time=1s)。
 黑名单 IP 拦截。
 交付物：超过频率请求直接返回 429。
📊 第四阶段：压测与优化 (第 6 周)
目标：用数据证明你的能力，这是面试的“证据”。

 4.1 编写压测脚本
 使用 JMeter 或 Wrk。
 模拟场景：1000 个用户同时抢 100 个库存。
 交付物：.jmx 压测脚本。
 4.2 执行压测并记录
 记录 优化前 的 QPS、响应时间、错误率。
 记录 优化后 (加缓存、加锁、加 MQ) 的数据。
 交付物：压测报告 (包含图表截图)。
 4.3 数据库优化
 开启 MySQL 慢查询日志。
 使用 EXPLAIN 分析慢 SQL，添加索引。
 交付物：无慢查询日志。
 4.4 故障演练 (可选但加分)
 模拟 Redis 宕机，系统是否降级？
 模拟 MQ 消息丢失，是否有补偿机制？
 交付物：异常处理逻辑代码。

💡 关键成功要素 (Checklist 中的 Checklist)
在执行过程中，请时刻检查以下几点，确保不跑偏：

代码是否开源？
 必须上传到 GitHub/Gitee，私有仓库面试官看不了。
是否有对比数据？
 没有压测对比图的高并发项目是没有灵魂的。
是否避免了过度设计？
 不要搞微服务拆分（单机单体即可），实习生项目拆分微服务会增加部署难度且无必要。
是否理解了每一行代码？
 尤其是 Redis 锁和 MQ 的部分，不要复制粘贴，要能手写核心逻辑。
是否有异常处理？
 锁失败了怎么办？MQ 消费失败了怎么办？这些边界情况才是面试重点。

高并发场馆预约系统 | 核心后端开发

针对热门场馆预约场景，设计并实现了基于 Spring Boot + Redis + RocketMQ 的高并发预约系统。
并发优化：通过 Redis 预减库存 + Lua 脚本 原子操作，将库存校验性能提升 10 倍；利用 Redisson 分布式锁 解决超约问题，锁粒度细化至“时段级”，并发支撑能力从 200 QPS 提升至 2000+ QPS。
稳定性保障：设计 滑动窗口限流算法 拦截恶意刷单请求；通过 MQ 异步削峰 解耦下单流程，保证高峰期接口响应时间低于 100ms。
数据一致性：采用 本地消息表 + 定时补偿 方案，确保预约记录与库存数据的最终一致性。
成果：项目已容器化部署，包含完整的 JMeter 压测报告 与 架构设计文档。
